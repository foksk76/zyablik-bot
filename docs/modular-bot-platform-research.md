# Task 11: Исследование модульной bot-platform для МАХ

Документ фиксирует рекомендуемый следующий шаг после принятия первого этапа проекта.

Это исследовательская задача. Она не меняет текущий Zabbix Webhook, не добавляет новый runtime и не расширяет принятую архитектуру без отдельного ADR.

## Краткая идея

Рассмотреть развитие проекта в сторону модульной bot-platform для МАХ, где:

```text
Zabbix = источник уведомлений
МАХ = транспорт доставки и канал входящих сообщений
Плагины = отдельные расширения возможностей бота
```

Первый прикладной сценарий:

```text
Пользователь или групповой чат отправляет сообщение боту МАХ.
Бот возвращает безопасную подсказку с chat_id или user_id.
Полученное значение используется для настройки получателя уведомлений в Zabbix.
```

## Почему это следующий шаг

Первый этап подтвердил базовую доставку уведомлений Zabbix -> МАХ.

Следующая полезная возможность — упростить получение `chat_id` / `user_id` для настройки получателей в Zabbix. Сейчас это остается ручным административным процессом.

При этом не требуется сразу строить большой bot-service. Нужен research, чтобы выбрать правильный путь:

- написать собственную минимальную платформу;
- адаптировать готовый open source bot/workflow framework;
- ограничиться отдельным identity-сервисом без полноценной платформы.

## Цель Task 11

Подготовить техническое решение для будущего MVP:

```text
MAX Identity Bot
```

MVP должен отвечать только на вопрос:

```text
Как безопасно получить chat_id или user_id из входящего сообщения МАХ и использовать это значение для настройки уведомлений в Zabbix?
```

## Не цель Task 11

В Task 11 не реализуются:

```text
отдельный промышленный bot-service
очередь сообщений
база данных
журнал доставки
автоматическая повторная отправка
управление Zabbix из МАХ
автоматическое изменение настроек Zabbix
обработка инцидентов из мессенджера
AI-логика
```

Если одно из этих требований станет обязательным, перед реализацией нужно создать отдельный ADR.

## Целевая модульная модель

Возможная будущая структура:

```text
bot-platform/
  core/
    event-router
    plugin-loader
    config
  transports/
    max/
  sources/
    zabbix/
  plugins/
    identity/
    delivery-log/
    retry/
    routing/
```

Назначение блоков:

| Блок | Назначение |
|---|---|
| `core` | общий роутинг событий, загрузка плагинов, конфигурация |
| `transports/max` | прием входящих событий МАХ и отправка ответов в МАХ |
| `sources/zabbix` | прием или нормализация событий Zabbix в будущих сценариях |
| `plugins/identity` | получение и возврат `chat_id` / `user_id` |
| `plugins/delivery-log` | будущий журнал доставки, если появится требование |
| `plugins/retry` | будущая повторная отправка, если появится требование |
| `plugins/routing` | будущая маршрутизация вне Zabbix, если появится требование |

## MVP: identity plugin

Первый плагин для оценки:

```text
plugins/identity
```

Минимальное поведение:

1. Принять входящее событие от МАХ.
2. Определить тип отправителя:
   - личный пользователь;
   - групповой чат.
3. Сформировать безопасный ответ:

```text
RecipientType: user_id
To: <user_id>
```

или:

```text
RecipientType: chat_id
To: <chat_id>
```

4. Вернуть пользователю краткую инструкцию для передачи ответственному за настройку Zabbix.
5. Не записывать токены, реальные идентификаторы и чувствительные значения в репозиторий.
6. Не менять настройки Zabbix автоматически.

## Пример ответа бота

Для личного диалога:

```text
Ваш идентификатор получателя для настройки уведомлений:

RecipientType: user_id
To: <user_id>

Передайте эти значения ответственному за настройку Zabbix Media type.
```

Для группового чата:

```text
Идентификатор группового чата для настройки уведомлений:

RecipientType: chat_id
To: <chat_id>

Передайте эти значения ответственному за настройку Zabbix Media type.
```

## Вариант A: собственная минимальная платформа

Суть:

```text
разработать небольшой self-hosted сервис с MAX transport и identity plugin
```

Плюсы:

- полный контроль над архитектурой;
- можно сделать минимальный функционал без лишних зависимостей;
- проще встроить требования безопасности и сопровождения;
- легче ограничить область ответственности.

Минусы:

- нужно самостоятельно реализовать прием входящих сообщений;
- нужно самостоятельно описать plugin lifecycle;
- потребуется собственная диагностика;
- выше ответственность за поддержку.

Когда выбирать:

```text
если готовые проекты слишком тяжелые, плохо поддерживаются, конфликтуют с требованиями безопасности или лицензией
```

## Вариант B: поиск готового open source проекта

Суть:

```text
найти self-hosted open source bot/workflow framework, куда можно добавить МАХ как транспорт, Zabbix как источник уведомлений и identity plugin для chat_id / user_id
```

Искать нужно не готовый проект именно под связку Zabbix + МАХ, а платформу с расширяемой архитектурой:

```text
transport / adapter architecture
plugin architecture
webhook receiver
HTTP API
self-hosted deployment
secret management outside repository
```

Возможные направления поиска:

```text
open source chatbot framework plugin backend adapter
open source chatops bot framework webhook backend
custom chat backend bot framework
self hosted workflow automation webhook bot
Zabbix webhook alert source bot framework
custom transport adapter chatbot framework
```

Кандидаты для исследования фиксируются только после проверки в открытых источниках.

## Вариант C: workflow-platform для прототипа

Суть:

```text
использовать self-hosted workflow-платформу как быстрый прототип для приема входящего события МАХ и возврата chat_id / user_id
```

Плюсы:

- быстрый прототип;
- удобно проверить сценарий без написания полноценной платформы;
- можно быстро показать ценность.

Минусы:

- не всегда подходит как промышленная bot-platform;
- сложнее контролировать архитектурную чистоту;
- возможны ограничения по лицензии, эксплуатации и безопасности;
- дальнейшее развитие может потребовать миграции на собственный сервис.

Когда выбирать:

```text
если цель — быстро проверить сценарий identity bot до принятия решения о промышленной реализации
```

## Критерии выбора подхода

| Критерий | Что проверить |
|---|---|
| Self-hosted | Можно ли развернуть внутри локального контура |
| Лицензия | Допустимо ли использовать в корпоративной среде |
| Расширяемость | Есть ли плагины, адаптеры или hooks |
| MAX transport | Можно ли добавить прием и отправку сообщений МАХ |
| Zabbix source | Можно ли принять webhook или событие от Zabbix |
| Identity use case | Можно ли безопасно вернуть `chat_id` / `user_id` |
| Секреты | Можно ли хранить токены вне репозитория |
| Логи | Можно ли вести диагностику без раскрытия чувствительных значений |
| Простота сопровождения | Понятна ли эксплуатация и обновление |
| Зависимости | Нет ли лишних компонентов для MVP |

## Краткая идея

Рассмотреть развитие проекта в сторону модульной bot-platform для МАХ, где:

```text
Zabbix = источник уведомлений
МАХ = транспорт доставки и канал входящих сообщений
Плагины = отдельные расширения возможностей бота
```

Первый прикладной сценарий:

```text
Пользователь или групповой чат отправляет сообщение боту МАХ.
Бот возвращает безопасную подсказку с chat_id или user_id.
Полученное значение используется для настройки получателя уведомлений в Zabbix.
```

## Почему это следующий шаг

Первый этап подтвердил базовую доставку уведомлений Zabbix -> МАХ.

Следующая полезная возможность — упростить получение `chat_id` / `user_id` для настройки получателей в Zabbix. Сейчас это остается ручным административным процессом.

При этом не требуется сразу строить большой промышленный bot-service. Второй этап должен выбрать путь для ограниченного MVP:

- написать собственную минимальную платформу;
- адаптировать готовый open source bot/workflow framework;
- ограничиться отдельным identity-сервисом без полноценной платформы.

По итогам Task 11.1 выбран путь адаптации open source bot framework: Hubot.

## Целевая модульная модель

Возможная будущая структура:

```text
bot-platform/
  core/
    event-router
    plugin-loader
    config
  transports/
    max/
  sources/
    zabbix/
  plugins/
    identity/
    delivery-log/
    retry/
    routing/
```

Назначение блоков:

| Блок | Назначение |
|---|---|
| `core` | общий роутинг событий, загрузка плагинов, конфигурация |
| `transports/max` | прием входящих событий МАХ и отправка ответов в МАХ |
| `sources/zabbix` | прием или нормализация событий Zabbix в будущих сценариях |
| `plugins/identity` | получение и возврат `chat_id` / `user_id` |
| `plugins/delivery-log` | будущий журнал доставки, если появится требование |
| `plugins/retry` | будущая повторная отправка, если появится требование |
| `plugins/routing` | будущая маршрутизация вне Zabbix, если появится требование |

## MVP: identity plugin

Первый плагин для оценки:

```text
plugins/identity
```

Минимальное поведение:

1. Принять входящее событие от МАХ.
2. Определить тип отправителя:
   - личный пользователь;
   - групповой чат.
3. Сформировать безопасный ответ:

```text
RecipientType: user_id
To: <user_id>
```

или:

```text
RecipientType: chat_id
To: <chat_id>
```

4. Вернуть пользователю краткую инструкцию для передачи ответственному за настройку Zabbix.
5. Не записывать токены, реальные идентификаторы и чувствительные значения в репозиторий.
6. Не менять настройки Zabbix автоматически.

## Пример ответа бота

Для личного диалога:

```text
Ваш идентификатор получателя для настройки уведомлений:

RecipientType: user_id
To: <user_id>

Передайте эти значения ответственному за настройку Zabbix Media type.
```

Для группового чата:

```text
Идентификатор группового чата для настройки уведомлений:

RecipientType: chat_id
To: <chat_id>

Передайте эти значения ответственному за настройку Zabbix Media type.
```

## Рассмотренные варианты

Подробное сравнение хранится в:

```text
docs/modular-bot-platform-candidates.md
```

### Вариант A: собственная минимальная платформа

Плюсы:

- полный контроль над архитектурой;
- можно сделать минимальный функционал без лишних зависимостей;
- проще встроить требования безопасности и сопровождения.

Минусы:

- нужно самостоятельно реализовать прием входящих сообщений;
- нужно самостоятельно описать plugin lifecycle;
- потребуется собственная диагностика;
- выше ответственность за поддержку.

Решение: не выбирать первым. Вернуться к варианту, если Hubot-path и Node-RED fallback окажутся нерациональными.

### Вариант B: open source bot framework

Проверены Hubot и Errbot.

Решение:

```text
Hubot — основной путь MVP.
Errbot — запасной кандидат для пересмотра.
```

Причины выбора Hubot:

- bot framework ближе к целевой модели, чем workflow-platform;
- MIT license;
- self-hosted;
- adapter model для chat providers;
- scripts model для расширения поведения;
- MVP можно ограничить identity-сценарием без базы данных, очереди, журнала доставки и Zabbix API.

### Вариант C: workflow-platform для прототипа

Проверены Node-RED и n8n.

Решение:

```text
Node-RED — fallback для workflow-прототипа.
n8n — отклонен для open source shortlist из-за source-available / Sustainable Use License.
```

Node-RED подходит для быстрой проверки HTTP identity-сценария, но не является целевой bot-platform.

## Критерии выбора подхода

| Критерий | Итог |
|---|---|
| Self-hosted | Hubot и Node-RED подходят |
| Лицензия | Hubot MIT; Node-RED Apache-2.0; Errbot GPL-3 требует оценки; n8n не проходит open source фильтр |
| Расширяемость | Hubot: adapters/scripts; Node-RED: custom nodes/flows |
| MAX transport | Для Hubot нужен custom adapter; для Node-RED возможен HTTP/custom node |
| Zabbix source | Для MVP Zabbix API не нужен; текущий Zabbix Webhook не меняется |
| Identity use case | Hubot и Node-RED подходят |
| Секреты | Токены должны храниться вне репозитория |
| Логи | Только обезличенная диагностика |
| Простота сопровождения | Hubot проще как bot framework; Node-RED проще для короткого прототипа |
| Зависимости | Hubot выбран для MVP, Node-RED только fallback |

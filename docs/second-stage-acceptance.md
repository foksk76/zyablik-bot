# Критерии завершения второго этапа

Документ фиксирует границу, стартовую задачу и критерии завершения второго этапа проекта.

Первый этап уже принят по `docs/project-acceptance.md`. Второй этап не пересматривает результат первого этапа и не меняет принятую схему доставки Zabbix -> МАХ через Zabbix Media type `Webhook`.

## Старт второго этапа

Второй этап начинается с Task 11:

```text
Task 11: Исследовать модульную bot-platform для МАХ и MVP получения chat_id / user_id
```

Рабочий документ Task 11:

```text
docs/modular-bot-platform-research.md
```

## Цель второго этапа

Выбрать технический подход для дальнейшего развития проекта в сторону модульной платформы взаимодействия с ботом МАХ.

Первый прикладной сценарий второго этапа:

```text
Пользователь или групповой чат отправляет сообщение боту МАХ.
Бот возвращает chat_id или user_id.
Полученное значение используется для настройки получателя уведомлений в Zabbix.
```

## Граница второго этапа

Второй этап является исследовательско-проектным.

Входит:

- исследование сценария получения `chat_id` / `user_id` через входящие сообщения бота МАХ;
- поиск open source кандидатов для модульной bot-platform или workflow-прототипа;
- сравнение вариантов: собственный сервис, готовый open source framework, workflow-platform;
- выбор рекомендуемого подхода для MVP `MAX Identity Bot`;
- подготовка ADR перед началом реализации нового сервиса, runtime или входящих webhooks;
- уточнение границ будущего MVP.

Не входит без отдельного ADR:

- реализация промышленного bot-service;
- запуск нового runtime;
- прием входящих webhooks от МАХ в продуктивном режиме;
- автоматическое изменение настроек Zabbix;
- интеграция с Zabbix API;
- очередь сообщений;
- база данных;
- журнал доставки;
- автоматическая повторная отправка;
- маршрутизация уведомлений вне Zabbix;
- обработка инцидентов из мессенджера;
- AI-логика.

## Критерии завершения второго этапа

Второй этап считается завершенным, когда выполнены условия:

- [ ] Описан и согласован целевой сценарий MVP `MAX Identity Bot`.
- [ ] Выполнен поиск в открытых источниках по open source bot/workflow framework кандидатам.
- [ ] По каждому кандидату зафиксированы:
  - назначение проекта;
  - лицензия;
  - активность разработки;
  - возможность self-hosted развертывания;
  - наличие plugin / adapter / webhook архитектуры;
  - возможность добавить МАХ как transport;
  - возможность принять Zabbix как источник уведомлений;
  - применимость для сценария получения `chat_id` / `user_id`;
  - ключевые риски.
- [ ] Подготовлена сравнительная таблица вариантов.
- [ ] Сделан вывод по подходу для MVP:
  - собственный минимальный сервис;
  - адаптация open source framework;
  - workflow-прототип;
  - отказ от реализации, если ценность не подтверждена.
- [ ] Подготовлен ADR на выбранный подход, если требуется новый сервис, runtime или входящие webhooks.
- [ ] Подтверждено, что текущий Zabbix Webhook не меняется в рамках второго этапа без отдельной задачи и ADR.
- [ ] Подтверждено, что первый этап остается принятым и не пересматривается.
- [ ] Выполнен `npm test`, если менялись файлы, покрываемые policy tests.

## Критерии качества выбора платформы

Выбранный подход должен быть оценен по критериям:

| Критерий | Описание |
|---|---|
| Security | секреты не хранятся в репозитории, реальные идентификаторы не публикуются |
| Self-hosted | решение можно развернуть в локальном контуре |
| Simplicity | MVP не требует лишних компонентов |
| Maintainability | понятны обновление, диагностика и сопровождение |
| Extensibility | есть способ добавлять плагины или модули |
| MAX compatibility | можно принять входящее сообщение МАХ и отправить ответ |
| Zabbix compatibility | можно использовать Zabbix как источник уведомлений без изменения принятого webhook-сценария |
| License | лицензия допустима для корпоративного применения |
| Operational risk | понятны риски эксплуатации и отказа |

## Правило перехода к реализации

Реализация начинается только после завершения исследования и принятия ADR, если выбранный подход добавляет новый компонент.

До ADR запрещено добавлять:

```text
новый сервис
новый runtime
обработчик входящих webhooks от МАХ
хранение состояния
журнал доставки
повторную отправку
маршрутизацию вне Zabbix
интеграцию с Zabbix API
```

## Итог второго этапа

Ожидаемый итог второго этапа:

```text
Подтвержден подход к развитию проекта после принятого первого этапа. Выбран или отклонен MVP MAX Identity Bot. Решение о реализации зафиксировано через ADR, если новый компонент требуется. Текущий Zabbix Webhook остается стабильным и не меняется без отдельного решения.
```

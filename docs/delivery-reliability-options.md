# Варианты повторной отправки и журнала доставки

Документ фиксирует возможное развитие после подтверждения базового сценария Zabbix -> МАХ.

Это не план немедленной реализации. Код webhook не меняется, новые компоненты не добавляются.

## Текущее состояние

На текущем этапе используется прямой Zabbix Webhook:

```text
Zabbix Action -> MAX Webhook -> MAX Bot API -> получатель МАХ
```

Подтверждено:

- тестовое сообщение доставлено;
- Problem доставлен;
- Recovery доставлен;
- параметры Media type описаны;
- получение `chat_id` / `user_id` описано как ручной административный процесс;
- повторное создание и перенос Media type описаны.

## Ограничения прямого webhook-подхода

Прямой webhook остается самым простым вариантом, но у него есть естественные ограничения:

- нет собственного постоянного журнала попыток доставки вне Zabbix;
- повторная отправка ограничена возможностями текущей схемы Zabbix и webhook runtime;
- сложная маршрутизация остается в правилах Zabbix и настройках получателей;
- входящие сообщения бота не обрабатываются;
- автоматизированная выдача `chat_id` / `user_id` пользователю не реализуется;
- диагностика доставки опирается на результат Zabbix Media type, Zabbix Action и ответ API.

Эти ограничения не мешают текущему подтвержденному сценарию доставки уведомлений.

## Вариант 1. Оставить прямой webhook без усложнения

Используется, если требуется только доставка уведомлений из Zabbix в МАХ.

Подходит, когда:

- список получателей небольшой;
- маршрутизация задается средствами Zabbix;
- достаточно стандартной диагностики Zabbix;
- не требуется хранить отдельную историю попыток доставки;
- не требуется автоматическая повторная отправка вне Zabbix;
- не требуется обработка входящих сообщений бота.

Решение на текущий этап: этот вариант остается базовым.

## Вариант 2. Регламентная повторная отправка без нового сервиса

Минимальное развитие без изменения архитектуры.

Идея:

- фиксировать ошибку доставки в Zabbix;
- проверять параметры `Token`, `RecipientType`, `To`, `HTTPProxy`;
- при необходимости повторять отправку через тест Media type или тестовый Action;
- результат фиксировать обезличенно в документации прогона.

Плюсы:

- не добавляет новых компонентов;
- не меняет webhook-код;
- подходит для пилота и ручной диагностики.

Минусы:

- не является автоматической повторной отправкой;
- не дает централизованного журнала попыток;
- зависит от ручных действий ответственного за настройку.

## Вариант 3. Журнал доставки в будущем сервисе

Идея:

- принимать событие от Zabbix в сервис;
- сохранять попытку доставки;
- фиксировать статус отправки в МАХ;
- хранить код ответа API и обезличенные диагностические поля;
- предоставлять простой способ диагностики последней попытки.

Возможные поля журнала:

```text
timestamp
event_id
recipient_type
recipient_hash
delivery_status
api_status_code
attempt_number
error_class
```

Чувствительные значения в журнале не должны храниться в открытом виде.

Такой вариант требует отдельного ADR до реализации.

## Вариант 4. Автоматическая повторная отправка

Идея:

- выполнять повторную отправку только для временных ошибок;
- ограничивать количество попыток;
- использовать паузу между попытками;
- не повторять отправку при ошибках настройки получателя или авторизации;
- фиксировать все попытки в журнале доставки.

Пример классов ошибок:

```text
temporary_error: можно повторить позже
configuration_error: требуется исправить настройки
authorization_error: требуется проверить токен
recipient_error: требуется проверить RecipientType и To
```

Такой вариант требует отдельного ADR до реализации.

## Вариант 5. Маршрутизация по группам получателей

Идея:

- хранить правила маршрутизации вне webhook-скрипта;
- отправлять разные категории уведомлений разным получателям;
- управлять маршрутами без изменения кода webhook;
- сохранять текущий webhook как простой транспортный компонент или заменить его сервисом после ADR.

Примеры признаков маршрутизации:

```text
severity
host group
service group
event tag
environment
maintenance window
```

На текущем этапе маршрутизация остается в Zabbix.

Такой вариант требует отдельного ADR, если правила выносятся из Zabbix в новый компонент.

## Вариант 6. Входящие сообщения бота для получения chat_id / user_id

Идея:

- пользователь или групповой чат отправляет сообщение боту;
- сервис принимает входящее событие;
- сервис возвращает безопасную подсказку с `chat_id` или `user_id`;
- полученное значение используется для дальнейшей настройки получателей уведомлений в Zabbix.

Этот вариант связан с MAX Identity Bot, а не с надежностью доставки Zabbix -> МАХ.

После ADR-0010 live-сценарий вынесен в Task 18. Он должен оставаться identity-only: получить входящее сообщение, вернуть `RecipientType` и `To`, не добавлять очередь, базу данных, журнал доставки или маршрутизацию уведомлений.

## Критерии перехода к реализации

Возвращаться к реализации новых механизмов стоит, если появится хотя бы одно подтвержденное требование:

- нужен постоянный журнал попыток доставки;
- нужна автоматическая повторная отправка;
- нужна обработка входящих сообщений бота;
- нужно автоматизированное получение `chat_id` / `user_id`;
- нужно централизованное управление маршрутами вне Zabbix;
- стандартной диагностики Zabbix недостаточно для сопровождения.

## Решение на текущий этап

Текущий webhook-код не меняется.

Очередь, база данных, журнал доставки, автоматическая повторная отправка и маршрутизация вне Zabbix не реализуются.

Описанные варианты считаются будущими опциями. Перед началом реализации любого из этих вариантов нужно создать ADR в `docs/decisions/`.

# Third stage stand and agent notes

Документ фиксирует проверку вариантов стенда для третьего этапа и правила применения Codex agent или аналога.

## Проверяемые варианты стенда

```text
1. WSL на Windows
2. LXC на Proxmox
```

Оба варианта допустимы для MVP, но решают разные задачи.

## WSL на Windows

Назначение:

```text
локальная разработка, быстрые unit-тесты, работа Codex agent / IDE / терминала на рабочей станции
```

Плюсы:

- быстро развернуть на Windows;
- удобно работать с Git, Node.js, npm и редактором;
- подходит для разработки каркаса, тестов и fixtures;
- WSL 2 позволяет запускать Linux-инструменты без отдельной виртуальной машины;
- systemd доступен в актуальных сценариях WSL и может использоваться для проверки service mode.

Ограничения:

- WSL не должен считаться промышленным стендом;
- systemd-сервисы сами по себе не гарантируют постоянное удержание WSL instance;
- входящие webhook-прогоны требуют отдельного решения по доступности endpoint;
- для проверки callback-сценария может потребоваться временный reverse tunnel или отдельный LXC-стенд.

Рекомендация:

```text
использовать WSL как основной developer sandbox
```

## LXC на Proxmox

Назначение:

```text
интеграционный стенд, длительный запуск сервиса, проверка сетевой доступности и service mode
```

Плюсы:

- ближе к серверной эксплуатации;
- можно выделить отдельный Linux container;
- проще держать сервис постоянно запущенным;
- удобнее проверять сетевую доступность входящего webhook endpoint;
- подходит для повторяемых прогонов.

Ограничения:

- требуется доступ к Proxmox и отдельный контейнер;
- нужно заранее определить сетевой сегмент, DNS/порт и правила доступа;
- если понадобится Docker-in-LXC, нужны дополнительные настройки или отдельная VM;
- для MVP предпочтительнее запускать Node.js напрямую без Docker, чтобы не усложнять LXC.

Рекомендация:

```text
использовать LXC как предпочтительный integration stand
```

## Предварительный выбор

| Задача | Предпочтительная среда |
|---|---|
| Разработка кода | WSL |
| Unit-тесты | WSL или LXC |
| Проверка fixtures | WSL или LXC |
| Длительный запуск сервиса | LXC |
| Проверка входящего webhook endpoint | LXC |
| Работа Codex agent / аналога | WSL или LXC, в зависимости от доступа |

## Минимальные требования к WSL-стенду

- Windows 10 2004+ или Windows 11.
- WSL 2.
- Linux distribution: Ubuntu или Debian.
- Node.js LTS.
- Git.
- Доступ к репозиторию.
- Локальный `.env` только вне commit.
- Возможность выполнить `npm test`.

## Минимальные требования к LXC-стенду

- Proxmox VE с поддержкой LXC.
- Debian или Ubuntu container.
- Доступ по SSH.
- Node.js LTS.
- Git.
- systemd service для длительного запуска, если требуется.
- Сетевой доступ к тестовому endpoint.
- Отдельная локальная конфигурация секретов вне репозитория.

## Режим работы Codex agent или аналога

Codex agent или аналог используется как исполнитель задач разработки, но не как автономный владелец архитектуры.

Разрешено:

- создавать каркас кода по Task 12.x;
- писать unit-тесты;
- запускать `npm test`;
- обновлять документацию;
- анализировать ошибки тестов;
- предлагать follow-up задачи.

Запрещено без отдельного решения:

- добавлять базу данных;
- добавлять очередь;
- добавлять журнал доставки;
- добавлять retry;
- подключать Zabbix API;
- менять текущий Zabbix Webhook;
- сохранять реальные секреты или идентификаторы;
- публиковать реальные callback URL.

# ADR-0001: Использовать явный AI-assisted каркас разработки

## Статус

Принято.

> **Примечание (обновлено):** Часть ссылок в этом ADR устарела.
> Задачи перенесены в `tasks/sprints/` (ADR-0004), а `scripts/verify-repo.sh` заменён на `npm test` (ADR-0004, ADR-0005).
> Актуальная структура `.agents/` описана в `AGENTS.md` и `.agents/README.md`.

## Дата

2026-07-07

## Контекст

Проект небольшой и решает прикладную задачу доставки уведомлений из Zabbix в МАХ. При этом проект планируется вести с участием AI-агентов и Codex.

Без явного контекста агент может начать расширять проект за пределы первого этапа: добавлять SIEM-интеграцию, AI-аналитику, отдельный backend, очередь, базу данных или хранение чувствительных параметров.

Нужно дать агентам понятные рамки: что проект делает сейчас, что не входит в этап, где лежит основной артефакт, как проверять изменения и где фиксировать решения.

## Решение

Использовать в репозитории явный AI-assisted каркас разработки:

- `AGENTS.md` — главные инструкции для Codex и других AI-агентов;
- `.agents/project-context.md` — короткий рабочий контекст для агента;
- `.agents/tasks/` — текущая задача и backlog;
- `.agents/checklists/` — обязательные проверки перед завершением задачи;
- `.agents/prompts/` — шаблоны постановки задач и review;
- `DEVELOPMENT.md` — процесс разработки;
- `scripts/verify-repo.sh` — базовая проверка репозитория;
- `docs/decisions/` — каноничное место для ADR.

## Рассмотренные альтернативы

### Только README

Плюсы:

- минимум файлов;
- проще читать человеку.

Минусы:

- README быстро становится перегруженным;
- агенту сложнее отделить пользовательское описание от правил разработки;
- нет отдельного места для задач, prompts и чек-листов.

Решение: отклонено.

### Только AGENTS.md

Плюсы:

- удобно для AI-агентов;
- меньше дублирования.

Минусы:

- не хватает структуры для задач, ADR и проверок;
- часть контекста теряется между разными типами работ.

Решение: отклонено.

### Явный AI-assisted каркас

Плюсы:

- агент быстрее входит в контекст;
- меньше риск случайно добавить лишнюю функциональность;
- проще проверять изменения;
- документация становится частью процесса разработки;
- прошлые решения остаются видимыми для будущих инженеров и агентов.

Минусы:

- появляется несколько служебных файлов;
- их нужно поддерживать в актуальном состоянии.

Решение: принято.

## Последствия

- AI-агенты должны начинать работу с `AGENTS.md`, `.agents/project-context.md` и текущей задачи.
- Существенные изменения процесса, архитектуры и границ проекта фиксируются через ADR в `docs/decisions/`.
- `.agents/` используется как рабочая область для агента, но не как основное хранилище проектных решений.
- Документация должна объяснять не только что сделано, но и почему выбран именно такой путь.

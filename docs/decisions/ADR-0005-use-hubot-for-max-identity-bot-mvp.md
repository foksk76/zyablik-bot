# ADR-0005: Использовать Hubot как основной вариант MVP MAX Identity Bot

## Статус

Принято.

## Дата

2026-07-07

## Контекст

Первый этап проекта принят: доставка уведомлений Zabbix -> МАХ через Zabbix Media type `Webhook` подтверждена и не меняется.

Второй этап начинается с исследования модульной bot-platform для МАХ. Первый прикладной сценарий второго этапа:

```text
входящее сообщение боту МАХ -> получение chat_id / user_id -> безопасная подсказка для настройки получателя в Zabbix
```

По Task 11 и Task 11.1 выполнено сравнение кандидатов. Исторические рабочие research-файлы удалены при cleanup документации; итоговое решение остается зафиксированным в этом ADR.

Рассматривались:

- Hubot;
- Errbot;
- Node-RED;
- n8n;
- Botpress;
- Mattermost.

## Решение

Для MVP `MAX Identity Bot` выбрать основной путь:

```text
Hubot-based MVP
```

Fallback-вариант для короткой проверки identity-сценария:

```text
Node-RED workflow-прототип
```

## Почему Hubot

Hubot выбран как основной вариант, потому что:

- это bot framework, а не общая workflow-platform;
- есть adapter model для chat providers;
- есть scripts model для расширения поведения;
- лицензия MIT проще для внутреннего корпоративного применения;
- можно ограничить MVP сценарием получения `chat_id` / `user_id`;
- не требуется менять текущий Zabbix Webhook;
- не требуется Zabbix API integration на MVP;
- не требуется база данных, очередь, журнал доставки или retry на MVP.

## Почему Node-RED оставлен как fallback

Node-RED не является целевой bot-platform, но подходит как быстрый workflow-прототип, если custom Hubot adapter для МАХ окажется дольше или рискованнее ожидаемого.

Fallback применяется только для проверки identity-сценария:

```text
incoming HTTP event -> extract chat_id / user_id -> response
```

Node-RED fallback не считается целевой промышленной архитектурой без отдельного решения.

## Граница MVP

Входит в будущий MVP:

- новый self-hosted runtime для identity-сценария;
- обработка входящих событий от МАХ;
- определение типа отправителя: пользователь или групповой чат;
- возврат безопасной подсказки с `RecipientType` и `To`;
- обезличенные логи диагностики;
- документация по запуску и проверке.

Не входит в MVP:

- изменение текущего `src/zabbix-media-type/max-webhook.js`;
- изменение существующего Zabbix Media type `MAX`;
- автоматическая настройка Zabbix через API;
- запись реальных `chat_id` / `user_id` в репозиторий;
- база данных;
- очередь сообщений;
- журнал доставки;
- автоматическая повторная отправка;
- маршрутизация уведомлений вне Zabbix;
- обработка инцидентов из МАХ;
- AI-логика.

## Требования к реализации после ADR

Перед написанием кода нужно создать отдельную задачу реализации MVP.

Минимальные требования к реализации:

- токен МАХ хранится только во внешней конфигурации или переменных окружения;
- реальные `chat_id` / `user_id` не попадают в репозиторий;
- текущий Zabbix Webhook остается без изменений;
- сервис не вызывает Zabbix API;
- сервис не хранит состояние, если это не будет отдельно обосновано;
- все примеры в документации обезличены;
- после изменений выполняется `npm test`, если изменяются файлы, покрываемые policy tests.

## Рассмотренные варианты

### Hubot

Плюсы:

- подходит как bot framework;
- MIT license;
- self-hosted;
- adapter / scripts model;
- ближе всего к идее модульной bot-platform.

Минусы:

- нужен custom MAX adapter или gateway;
- появляется новый runtime;
- нужна отдельная эксплуатационная документация.

Решение: принят как основной путь MVP.

### Errbot

Плюсы:

- plugin/backend architecture;
- webhook support;
- self-hosted;
- хорошо подходит концептуально.

Минусы:

- GPL-3 требует отдельной лицензионной оценки;
- нужен custom MAX backend;
- активность ниже, чем у Node-RED и n8n.

Решение: оставить как запасной кандидат для пересмотра, но не выбирать первым.

### Node-RED

Плюсы:

- Apache-2.0;
- self-hosted;
- быстро проверяет HTTP/webhook identity-сценарий;
- низкий порог прототипирования.

Минусы:

- не является bot framework;
- при росте логики сложнее сохранять архитектурную чистоту;
- нужен контроль доступа к редактору и flow-конфигурациям.

Решение: оставить fallback-вариантом для workflow-прототипа.

### n8n

Плюсы:

- self-hosted;
- webhook и HTTP nodes;
- custom/community nodes.

Минусы:

- source-available / Sustainable Use License, не проходит open source фильтр;
- runtime избыточен для простого identity-MVP;
- требуется лицензионная оценка.

Решение: отклонено для open source shortlist.

### Botpress

Плюсы:

- есть SDK/CLI;
- MIT для текущего репозитория;
- развитая bot/agent экосистема.

Минусы:

- текущий публичный путь ориентирован на Botpress Cloud;
- deploy интеграций завязан на workspace / hub;
- plugins в текущем README отмечены как coming soon;
- on-prem v12 выглядит legacy для нового MVP.

Решение: отклонено.

### Mattermost

Плюсы:

- self-hosted;
- есть webhooks, plugins и bot accounts;
- подходит для интеграций внутри Mattermost.

Минусы:

- это отдельная chat-platform, а не transport для МАХ;
- добавляет лишний коммуникационный контур;
- не решает задачу получения МАХ `chat_id` / `user_id`.

Решение: отклонено.

## Последствия

- Второй этап получает выбранный путь для MVP.
- Текущий Zabbix Webhook остается стабильным.
- Реализация MVP не начинается без отдельной задачи.
- Любое расширение за пределы identity-сценария требует нового ADR или отдельной задачи с явной границей.
- Node-RED может использоваться только как fallback-прототип, если Hubot-path окажется нерациональным для короткой проверки.

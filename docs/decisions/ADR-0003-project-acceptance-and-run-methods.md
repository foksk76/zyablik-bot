# ADR-0003: Выделить критерии завершения проекта и методы прогонов

## Статус

Принято.

## Дата

2026-07-07

## Контекст

В проекте появились несколько уровней критериев:

- критерии завершения проекта в целом;
- критерии завершения отдельных задач;
- checkpoints в плане работ;
- проверки репозитория;
- ручные и интеграционные прогоны.

Если хранить project-level критерии в нескольких местах, AI-агенты могут начать трактовать границы первого этапа по-разному и дублировать или расширять требования.

Также для задач нужен единый порядок: какой метод работы применяется, какой прогон выполняется и какой skill использовать.

## Решение

Создать единый документ project-level приемки:

```text
docs/project-acceptance.md
```

В нем хранить только условия завершения первого этапа проекта в целом.

Критерии отдельных задач оставить в:

```text
tasks/sprints/
```

План фаз, зависимости и checkpoints оставить в:

```text
tasks/sprints/
```

Методы разработки, тестирования и прогонов описывать в:

```text
DEVELOPMENT.md
```

Для задач в `tasks/sprints/` явно указывать:

- Method;
- Skill;
- Acceptance criteria;
- Verification;
- Dependencies;
- Files likely touched;
- Estimated scope.

Код менять только на основании документации проекта, внешней документации или отдельного ADR. Если поведение среды выполнения или API не подтверждено, сначала создается задача на уточнение или ADR, а не код.

## Рассмотренные альтернативы

### Хранить критерии завершения в README

Плюсы:

- легко найти человеку;
- один файл на входе.

Минусы:

- README быстро перегружается;
- критерии начинают смешиваться с описанием проекта;
- агенты чаще копируют и дублируют эти условия в другие документы.

Решение: отклонено.

### Хранить критерии завершения в tasks/sprints/

Плюсы:

- рядом с планом работ;
- удобно для исполнителя.

Минусы:

- смешиваются project-level критерии и task-level критерии;
- при изменении задач можно случайно изменить границы проекта.

Решение: отклонено.

### Хранить критерии завершения в отдельном документе docs/project-acceptance.md

Плюсы:

- одно каноничное место для приемки;
- README и tasks остаются легче;
- AI-агенту проще отличить завершение проекта от завершения задачи;
- границы первого этапа не размываются.

Минусы:

- нужен отдельный документ и ссылки на него.

Решение: принято.

## Последствия

- `docs/project-acceptance.md` становится единственным источником project-level критериев завершения первого этапа.
- `tasks/sprints/` хранит критерии конкретных задач, порядок выполнения, зависимости, риски и checkpoints без дублирования project-level приемки.
- `DEVELOPMENT.md` хранит методы разработки, тестирования, прогонов и соответствующие skills.
- Изменение project-level критериев требует нового ADR или обновления этого решения.

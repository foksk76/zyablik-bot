# ADR-0004: Использовать Node.js policy tests и GitHub Actions вместо bash-проверки

## Статус

Принято.

## Дата

2026-07-07

## Контекст

В проекте использовался shell-скрипт `scripts/verify-repo.sh` для проверки структуры репозитория и базовых правил документации.

Скрипт проверял:

- наличие обязательных файлов;
- хранение ADR только в `docs/decisions/`;
- хранение задач только в `tasks/`;
- отсутствие старых организационных формулировок в документации и задачах.

Такой подход работал локально, но имел ограничения:

- проверка была завязана на bash;
- правила было сложнее развивать как набор независимых тестов;
- GitHub не запускал проверку автоматически;
- для будущих статических проверок webhook и параметров Media type удобнее использовать тестовый runner.

## Решение

Перенести проверки из `scripts/verify-repo.sh` в Node.js policy tests на базе встроенного test runner.

Использовать команду:

```bash
npm test
```

как основной локальный и CI-прогон.

Добавить GitHub Actions workflow:

```text
.github/workflows/verify.yml
```

Workflow запускает `npm test` на `push`, `pull_request` и вручную через `workflow_dispatch`.

После переноса покрытия удалить:

```text
scripts/verify-repo.sh
```

## Граница решения

Это решение меняет только механизм проверки репозитория.

Не меняется:

- архитектура webhook;
- логика отправки Zabbix -> МАХ;
- порядок ручных интеграционных прогонов;
- необходимость отдельного ADR для Zabbix container, test harness или нового runtime, который эмулирует Zabbix.

## Рассмотренные варианты

### Оставить только bash script

Плюсы:

- просто;
- не требует Node.js тестов.

Минусы:

- нет автоматического запуска в GitHub;
- сложнее развивать точечные проверки;
- неудобно проверять параметры Media type и webhook как отдельные тесты.

Решение: отклонено.

### GitHub Actions запускает bash script

Плюсы:

- минимальное изменение;
- можно быстро получить CI.

Минусы:

- bash остается основным местом правил;
- сложнее развивать проверки как тестовый набор;
- часть проверок webhook и документации удобнее писать на JavaScript.

Решение: отклонено в пользу полной миграции.

### Node.js policy tests + GitHub Actions

Плюсы:

- один способ проверки локально и в CI;
- проверки можно разделять по файлам;
- можно добавить статические тесты параметров Media type и webhook;
- не нужен отдельный shell-скрипт.

Минусы:

- нужен Node.js runtime для проверки;
- правила проверки нужно поддерживать как тестовый код.

Решение: принято.

## Последствия

- `npm test` становится основной командой проверки репозитория.
- `npm run verify` остается алиасом на `npm test`.
- GitHub Actions запускает проверку автоматически.
- `scripts/verify-repo.sh` удаляется.
- Новые проверки добавляются в `tests/` как отдельные test-файлы.
- Если потребуется интеграционный runtime, Zabbix container или test harness, для этого нужен отдельный ADR.

# ADR-0021: Обработка `bot_started` событий с приветствием

## Статус

Принято.

## Дата

2026-07-16

## Контекст

ADR-0020 расширил `REPLY_UPDATE_TYPES` на `['message_created', 'bot_added']` и решил не обрабатывать `bot_started` (личный контекст, навязчивое приветствие). Однако баг-репорт показал, что пользователи ожидают приветственное сообщение при начале диалога с ботом (direct message). Текущее поведение — `bot_started` игнорируется — не соответствует ожиданиям.

**Связь с ADR-0020:** Этот ADR частично заменяет решение ADR-0020 по обработке `bot_started` — `bot_added` по-прежнему обрабатывается как в ADR-0020, но `bot_started` теперь тоже обрабатывается вместо того, чтобы игнорироваться.

## Решение

Расширить `REPLY_UPDATE_TYPES` на `['message_created', 'bot_added', 'bot_started']`.

### Обработка `bot_started` в pipeline

`bot_started` обрабатывается аналогично `bot_added` — отправляет приветственное сообщение «Ready to help.» без вызова command dispatch.

```text
normalize → updateType === 'bot_added' || updateType === 'bot_started'?
  → да:  send(welcomeMessage) — return
  → нет: parseCommand → command dispatch → send
```

### Приветственное сообщение

```text
Ready to help.
```

Тот же текст, что и для `bot_added`.

### Формат ответа

Pipeline формирует text-ответ (ADR-0019) с `event.recipient`:

```js
{
  kind: 'text',
  text: 'Ready to help.',
  recipient: event.recipient
}
```

Для `bot_started` `event.recipient` будет `{ kind: 'user', value: user_id }`.

## Почему `bot_started` теперь обрабатывается

- Пользователи ожидают приветствие при первом контакте с ботом (личный диалог).
- Отсутствие приветствия выглядит как неисправность бота.
- Краткое приветствие не перегружает интерфейс.

## Почему не вызывать `/help` автоматически

- `/help` выводит список команд — это информационный ответ, не приветствие.
- Автоматический вывод `/help` при начале диалога перегружает интерфейс.
- «Ready to help.» — достаточно, чтобы пользователь знал: бот работает и готов к командам.

## Последствия

- `REPLY_UPDATE_TYPES` расширяется с 2 на 3 элемента
- `live-pipeline.js` и `dry-run-pipeline.js` получают дополнительную проверку `updateType === 'bot_started'`
- Существующие тесты обновлены: `bot_started` теперь возвращает приветствие
- `docs/live-identity-bot.md` обновлено

## Рассмотренные альтернативы

### Оставить `bot_started` игнорируемым

Минус: не соответствует ожиданиям пользователей, бот кажется неработоспособным.

### Автоматический `/help` при `bot_started`

Минус: перегружает интерфейс, длинный вывод при первом контакте.

### Обработка `bot_started` через отдельный pipeline

Минус: дублирование HTTP-логики (normalize, send). Избыточно для одного дополнительного update type.

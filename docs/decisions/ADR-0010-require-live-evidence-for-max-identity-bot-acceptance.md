# ADR-0010: Требовать live evidence для приемки MAX Identity Bot

## Статус

Принято.

## Дата

2026-07-09

## Контекст

В `docs/project-acceptance.md` есть project-level сценарий:

```text
бот МАХ возвращает user_id / chat_id для настройки получателя в Zabbix
```

После Task 12-14 в репозитории есть bot-platform, dry-run pipeline, synthetic fixtures и safe test bot для outbound-only LXC. Эти артефакты полезны для разработки и проверки формата ответа, но они не доказывают, что реальный бот МАХ:

- получает живое входящее сообщение от пользователя или из чата;
- вызывает реальный MAX API для отправки ответа;
- возвращает пользователю видимые `user_id` / `chat_id`;
- корректно работает с секретами и без записи реальных идентификаторов в репозиторий.

Пользовательский симптом "сообщение отправлено боту, но не прочитано и нет ответа" показывает, что dry-run evidence недостаточно для приемки live-сценария.

## Решение

Считать критерий `бот МАХ возвращает user_id / chat_id` выполненным только после live-проверки с реальным MAX Bot API.

Для приемки live-сценария нужны:

- подтвержденный режим получения входящих событий от МАХ: `long_polling` или `webhook`;
- реализация сетевого inbound path на основании документации MAX Bot API;
- реализация сетевого outbound response path на основании документации MAX Bot API;
- реальное входящее сообщение от пользователя и видимый ответ бота с `RecipientType: user_id` и обезличенным `To`;
- реальное входящее сообщение из группового чата или другого поддержанного chat-сценария и видимый ответ бота с `RecipientType: chat_id` и обезличенным `To`;
- обезличенный test-run документ с датой, режимом транспорта, командами проверки, результатом и подтверждением, что реальные токены и идентификаторы не попали в репозиторий.

Dry-run, synthetic fixtures и safe test bot остаются допустимыми доказательствами разработки, но не являются достаточными доказательствами live-приемки.

Признак "сообщение помечено прочитанным" не является обязательным критерием до подтверждения официального метода MAX Bot API для read/ack. Если такой метод есть и будет использоваться, его нужно добавить отдельной задачей или ADR.

## Рассмотренные альтернативы

### Оставить dry-run evidence достаточным

Плюсы:

- не требуется новая интеграционная проверка;
- текущие документы остаются формально закрытыми.

Минусы:

- не подтверждается реальный пользовательский сценарий;
- можно принять бота, который не получает и не отправляет живые сообщения.

Решение: отклонено.

### Требовать read/ack как обязательную часть приемки

Плюсы:

- пользователь видит, что сообщение обработано.

Минусы:

- наличие и семантика read/ack должны быть подтверждены официальной документацией MAX Bot API;
- нельзя добавлять это требование догадкой.

Решение: отклонено до подтверждения API.

### Требовать live response, но не read/ack

Плюсы:

- проверяет главный пользовательский результат: бот отвечает `user_id` / `chat_id`;
- не додумывает неподтвержденное поведение API;
- сохраняет границу MVP без базы данных, очереди и retry.

Минусы:

- нужен отдельный live test-run в среде с реальным MAX Bot API.

Решение: принято.

## Последствия

- `docs/project-acceptance.md` должен явно отделять live evidence от dry-run evidence.
- Финальная приемка live identity-сценария невозможна без обезличенного live test-run.
- Реальные токены, `user_id`, `chat_id`, внутренние URL и организационные названия не записываются в репозиторий.
- Текущий Zabbix Webhook остается без изменений.
- Очередь, база данных, журнал доставки, retry и управление Zabbix из МАХ по-прежнему не входят в приемку без отдельного ADR.

# ADR-0013: Использовать многоуровневую архитектуру safe logger для маскировки секретов

## Статус

Принято.

## Дата

2026-07-15

## Контекст

Bot-platform обрабатывает HTTP-запросы с токенами доступа (`maxBotToken`), параметрами авторизации и идентификаторами пользователей. Логи попадают в systemd journal и могут содержать:

- Токены в заголовках `Authorization`
- Ответы API с `user_id` и `chat_id`
- Конфигурацию с секретами

Стандартный `console.log()` не маскирует секреты. Один забытый `console.log(response.text)` в продакшене утечёт токен бота в systemd journal.

## Решение

Использовать `createSafeLogger()` как единственный способ логирования во всех модулях. Многоуровневая маскировка:

### Уровень 1: Замена значений (value-level)

Все известные секреты заменяются на `[redacted]`. Источники секретов:
- Явный список `options.secrets` (массив строк)
- Конфигурация: ключи из `CONFIG_SECRET_KEYS` (`maxBotToken`, `token`, `secret`, `password`, `authorization`, `apiKey`, `apiToken`)

### Уровень 2: Детектирование по ключам (key-pattern)

Любой ключ объекта, совпадающий с `SENSITIVE_KEY_PATTERN` (`/(token|secret|password|authorization|api[-_]?key|cookie)/i`), заменяется на `[redacted]` независимо от значения.

### Уровень 3: Рекурсивная маскировка объектов

`maskValue()` рекурсивно обходит массивы и объекты, применяя все уровни маскировки. Циклические ссылки защищены через `WeakSet`.

### Интерфейс

```js
const logger = createSafeLogger({ config: runtimeConfig, secrets: [token] });
logger.info('response received', { body: response.body });
// body.token автоматически заменён на [redacted]
```

## Почему не console напрямую

- `console.log()` не маскирует секреты
- Нет способа гарантировать, что все вызовы пройдут через маскировку
- Агенты могут случайно использовать `console` вместо `logger`

## Почему не библиотечный решение (winston, pino)

- Нулевая зависимость (ADR-0015) — لا يمكن использовать npm-пакеты
- Стандартные библиотеки не имеют встроенной маскировки секретов
- Собственная реализация = 70 строк, полностью контролируемая

## Почему не context-based redaction (только в context, не в message)

- Секреты могут попасть в message через интерполяцию (`${token}`)
- Маскировка только context недостаточна

## Последствия

- Все модули используют `createSafeLogger()` вместо `console`
- Тесты проверяют отсутствие секретов в логах: `doesNotMatch(serialized, /synthetic-secret-token/)`
- Добавление нового модуля требует передачи `logger` через options
- Новые конфигурационные ключи с секретами добавляются в `CONFIG_SECRET_KEYS`

## Рассмотренные альтернативы

### Библиотечный logger (winston/pino)

Минус: нарушает ADR-0015 (нулевые зависимости), не решает проблему маскировки.

### Простая замена по regex

Минус: не маскирует объекты, не защищает от циклических ссылок, легко пропустить ключ.

### Interceptor pattern (перехват所有 console方法)

Минус: неявный, сложно тестировать, не гарантированно покрывает все вызовы.

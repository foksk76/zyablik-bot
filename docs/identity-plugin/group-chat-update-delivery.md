# Group chat update delivery: official source confirmation

Документ подтверждает в открытых источниках MAX Bot API, как и при каких условиях бот получает `message_created` в групповом чате. Закрывает Sprint 5 Task 5.1.

## Status

```text
Confirmed by official MAX Bot API documentation (dev.max.ru).
```

Checked on: 2026-07-14.

## Вопрос

Live-диагностика показала, что бот, добавленный в групповой чат как участник, не получает `message_created`: прямой запрос `GET /updates` с реальным токеном возвращал `HTTP 200 {"updates": [], "marker": ...}`. Нужно было подтвердить в открытых источниках, при каких условиях бот получает события из группового чата.

## Авторитетные источники

Source type: официальная публичная документация MAX Bot API.

Official source URLs (проверены напрямую, HTTP 200):

- `https://dev.max.ru/docs-api/objects/Update` — объект `Update`, перечень типов событий.
- `https://dev.max.ru/docs-api/methods/GET/updates` — метод Long Polling, параметры и описание результата.
- `https://dev.max.ru/docs-api/methods/POST/subscriptions` — метод подписки на Webhook, тело запроса, `update_types`.

Дополнительный контекст из того же источника:

- `https://dev.max.ru/docs-api` — обзор API, общий баннер про сертификат Минцифры.

## Подтверждённый ответ

### Корневое условие доставки updates из группового чата

Из описания результата метода `GET /updates` (поле `updates`):

> Список обновлений о событиях в чатах и каналах, **в которые добавлен бот**. **Обратите внимание, чтобы получать события из групповых чатов и каналов, бот должен быть администратором.** Подробнее о возможных событиях — в описании объекта Update.

Source: `https://dev.max.ru/docs-api/methods/GET/updates` (раздел «Результат → updates»).

Это объясняет live-наблюдение: бот был добавлен в групповой чат как обычный участник, а не администратор, поэтому MAX не доставлял ему `message_created` и `GET /updates` возвращал пустой список `updates`.

### Тип события message_created

Из описания объекта `Update` (раздел «Типы событий»):

> `message_created` — пользователь отправил новое сообщение или опубликовал пост

Source: `https://dev.max.ru/docs-api/objects/Update`.

### Фильтрация по типам событий

И Long Polling, и Webhook принимают список типов событий, которые бот хочет получать.

Long Polling — параметр `types` запроса `GET /updates`:

> `types` `string[]` Nullable optional. Пример: `types=message_created,message_callback`. Список типов событий, которые вы хотите получать, например: `message_created`, `message_callback`. Полный список возможных событий смотрите в описании объекта Update.

Source: `https://dev.max.ru/docs-api/methods/GET/updates` (раздел «Параметры → types»).

Webhook — поле `update_types` тела `POST /subscriptions`:

> `update_types` `string[]` optional. Пример: `["message_created", "bot_started"]`. Список типов событий, которые хочет получать ваш бот.

Source: `https://dev.max.ru/docs-api/methods/POST/subscriptions` (раздел «Тело запроса → update_types»).

Таким образом, условие доставки складывается из двух независимых требований:

1. бот является **администратором** данного группового чата/канала;
2. тип события `message_created` входит в запрошенный ботом список `types` (Long Polling) или `update_types` (Webhook).

Только второе условие у нас уже выполнено: `MAX_POLL_TYPES` по умолчанию содержит `message_created` (`src/bot-platform/core/config.js`, `DEFAULT_MAX_POLL_TYPES`).

### Что НЕ является условием доставки

В официальной документации не указано, что для доставки `message_created` боту нужно упоминание (`@bot`) или ответ (reply) на сообщение бота. Условие одно — бот-администратор. Гипотеза про «упоминание/reply», высказанная на раннем этапе диагностики, **официальной документацией не подтверждена и снимается**.

## Связанная находка: сертификат Минцифры

На всех страницах `dev.max.ru/docs-api` размещён общий баннер:

> ⚠️ До 19 июля 2026. Для корректной работы ваших чат-ботов и мини-приложений необходимо перенаправить HTTP-запросы с домена `platform-api.max.ru` на `platform-api2.max.ru`, **а также добавить сертификат Минцифры в список доверенных.**

Source: `https://dev.max.ru/docs-api/methods/GET/updates` (баннер в верхней части страницы; повторяется на всех страницах API).

Это официальное подтверждение TLS/CA-настройки live-окружения проекта: необходимость доверять сертификату Минцифры (`Russian Trusted Root CA`) документирована MAX. В live-окружении это решено переменной `NODE_EXTRA_CA_CERTS` (см. `docs/live-identity-bot.md`).

## Что не подтверждено

- Не найдена отдельная официальная страница с детальным описанием прав бота-администратора в чате (полный список того, что администратор может/не может). Для identity-сценария это не требуется — достаточно условия «бот должен быть администратором».
- Не подтверждено отдельно, отличается ли поведение каналов (`is_channel: true`) от групповых чатов в плане доставки `message_created`. Документация формулирует условие единообразно: «групповых чатов и каналов».

## Следствия для проекта

- **Task 4.2** (live chat `chat_id` verification) остаётся открытой не из-за дефекта кода, а из-за конфигурации чата: бот должен быть назначен администратором тестируемого группового чата, после чего `GET /updates` начнёт доставлять `message_created`.
- Код identity-бота менять не нужно: он уже запрашивает `message_created` и корректно обрабатывает входящие update после фикса бага №1 (PR #2 — бот отвечает только на `message_created`).
- Операционное действие для закрытия Task 4.2: назначить бота администратором целевого чата и повторить live-прогон.

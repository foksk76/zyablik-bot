# Критерии завершения третьего этапа

Документ фиксирует границы и критерии завершения третьего этапа проекта.

Третий этап начинается после принятого второго этапа и реализует MVP `MAX Identity Bot` на основе ADR-0005.

Статус после ADR-0010: этот документ описывает dry-run/safe-test MVP bot-platform. Live-приемка реального MAX Identity Bot вынесена в `docs/project-acceptance.md`, `docs/live-identity-bot.md` и Task 18.

```text
ADR: docs/decisions/ADR-0005-use-hubot-for-max-identity-bot-mvp.md
Основной путь: Hubot-based MVP
Fallback: Node-RED workflow-прототип
```

## Цель третьего этапа

Реализовать минимальную модульную bot-platform для МАХ и проверить identity-сценарий без хранения чувствительных данных:

```text
synthetic или тестовое входящее событие -> определение идентификатора получателя -> ответ с параметрами для Zabbix
```

Платформа должна быть готова к последующему расширению через плагины, но третий этап ограничен identity-сценарием.

## Граница третьего этапа

Входит:

- каркас модульной bot-platform;
- контракты транспорта МАХ для приема входящих событий;
- контракты транспорта МАХ для отправки ответов;
- identity plugin для определения типа получателя;
- обезличенные примеры входящих событий;
- локальный test harness;
- документация по запуску стенда;
- проверка стенда в WSL и/или LXC;
- описание применения Codex agent или аналога;
- unit-тесты и CI-прогон.

Не входит без отдельного ADR:

- изменение текущего Zabbix Media type `MAX`;
- изменение `src/zabbix-media-type/max-webhook.js`;
- автоматическая настройка Zabbix;
- хранение реальных идентификаторов получателей в репозитории;
- база данных;
- очередь сообщений;
- журнал доставки;
- автоматическая повторная отправка;
- маршрутизация уведомлений вне Zabbix;
- обработка инцидентов из мессенджера;
- AI-логика внутри runtime бота;
- промышленный эксплуатационный режим.
- live-приемка MAX Identity Bot без отдельного test-run по ADR-0010.

## Минимальная целевая структура

```text
bot-platform/
  core/
    config
    logger
    plugin-loader
    event-router
  transports/
    max/
      inbound-webhook
      outbound-client
      event-normalizer
  plugins/
    identity/
      handler
      formatter
  test-fixtures/
    max-inbound-user.json
    max-inbound-chat.json
```

## Acceptance criteria

- [x] Создан каркас модульной bot-platform.
- [x] Реализован MAX inbound transport contract для dry-run/safe-test path.
- [x] Реализован MAX outbound transport contract для dry-run/safe-test path.
- [x] Реализован identity plugin.
- [x] Для личного сообщения бот формирует параметры получателя для Zabbix.
- [x] Для группового чата бот формирует параметры получателя для Zabbix.
- [x] Секреты и реальные идентификаторы не попадают в репозиторий.
- [x] Конфигурация вынесена во внешнее окружение или локальный файл, не предназначенный для коммита.
- [x] Есть обезличенные fixtures для входящих событий МАХ.
- [x] Есть локальный прогон обработки fixtures без обращения к реальному API.
- [x] Есть интеграционный прогон с тестовым ботом или явно зафиксировано, почему он отложен.
- [x] Проверена возможность запуска стенда в WSL или LXC.
- [x] Описан режим работы Codex agent или аналога со стендом.
- [x] `npm test` проходит локально или в CI.
- [x] GitHub Actions green после изменений третьего этапа.
- [x] Текущий Zabbix Webhook не изменен.

## Live Gap

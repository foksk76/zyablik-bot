# Development guide

Этот файл описывает, как вести разработку проекта с участием человека и AI-агентов.

## Рабочий цикл

1. Проверить границы проекта в `AGENTS.md`, `.agents/project-context.md` и `docs/project-context.md`.
2. Проверить прошлые решения в `docs/decisions/README.md`.
3. Проверить план работ в `tasks/plan.md`.
4. Выбрать задачу из `tasks/todo.md`.
5. Внести минимальное изменение.
6. Обновить документацию, если изменилось поведение.
7. Обновить ADR, если изменилось архитектурное решение, процесс или граница проекта.
8. Запустить проверку:

```bash
bash scripts/verify-repo.sh
```

9. Оформить результат в PR или коротком описании изменений.

## Основные файлы

```text
README.md                                общее описание проекта
AGENTS.md                                правила для Codex и AI-агентов
.agents/project-context.md               краткий рабочий контекст для агента
docs/project-context.md                  полный контекст проекта
docs/documentation-policy.md             правила ведения документации
docs/decisions/                          ADR и история решений
tasks/plan.md                            план работ по phases/checkpoints
tasks/todo.md                            исполняемый список задач
src/zabbix-media-type/max-webhook.js     основной webhook-скрипт
docs/zabbix-media-type.md                описание настройки Zabbix Media type
scripts/verify-repo.sh                   базовая проверка репозитория
```

## Изменение webhook-скрипта

Если меняется логика `src/zabbix-media-type/max-webhook.js`, обязательно проверить:

- не поменялся ли формат ожидаемых параметров Zabbix;
- не появились ли реальные значения параметров;
- не сломана ли отправка Problem/Recovery;
- обновлен ли `docs/zabbix-media-type.md`;
- нужен ли новый ADR в `docs/decisions/`.

## Задачи

Задачи ведутся только в:

```text
tasks/plan.md
tasks/todo.md
```

Новая задача должна иметь:

- описание;
- acceptance criteria;
- verification;
- dependencies;
- files likely touched;
- estimated scope.

`.agents/` используется как рабочий контекст агента, но не как хранилище задач.

## Архитектурные изменения

Если задача требует отдельного сервиса, очереди, базы данных, новой интеграции или изменения границ этапа, сначала создать ADR в `docs/decisions/`.

ADR должен объяснять:

- контекст;
- принятое решение;
- рассмотренные альтернативы;
- последствия.

## Документация

Документация ведется по `docs/documentation-policy.md`.

Короткое правило: код показывает что сделано, документация объясняет почему это сделано именно так.

## Стиль

Документация пишется по-русски, без лишнего маркетинга. Формулировки должны быть понятны инженерам, администраторам мониторинга и сопровождающим сервисов.
